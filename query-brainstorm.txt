  O problema real e: a query vai crua para embedding. Mas nem toda melhoria

  Tres camadas de melhoria, em ordem de custo

  1. Engenharia pura (zero LLM, zero latencia adicional)

  - Extracao de metadados por regex -- "dano moral 2023" -> filtro
  data_publicacao LIKE '2023%' extraido por pattern matching. Numeros de
  processo, classes (REsp, HC, AREsp), anos, nomes de ministros -- tudo
  extraivel com regex
  - Tuning de RRF -- o k do RRF (tipicamente 60) muda drasticamente os
  resultados. Nunca foi tuneado
  - Pesos dense vs sparse -- a fusao trata ambos igualmente. Para queries curtas
   e tecnicas, sparse deveria pesar mais
  - Stopwords juridicas -- remover termos como "jurisprudencia", "entendimento",
   "STJ" da query antes do embedding (sao meta-linguagem, nao conteudo)

  2. Expansao estatica (dicionario, sem LLM)

  - Thesaurus juridico -- mapa pre-construido: "dano moral" -> tambem buscar
  "responsabilidade civil", "indenizacao". "CDC" -> "Codigo de Defesa do
  Consumidor", "Lei 8.078"
  - Multi-query por reformulacao sintatica -- gerar variantes automaticas: "dano
   moral em contrato bancario" -> ["contrato bancario dano moral",
  "responsabilidade banco dano moral"]

  3. LLM no loop (latencia +1-3s, custo por query)

  - Decomposicao multi-perspectiva -- so um LLM consegue fazer isso bem
  - Step-back prompting -- abstrair para principios juridicos
  - Intent detection -- entender se o usuario quer jurisprudencia, legislacao,
  doutrina

  Minha avaliacao revisada

  A camada 1 e a que tem melhor relacao custo/beneficio. E puro codigo, sem
  dependencia externa, sem latencia adicional. A extracao de metadados por regex

  opt-in ("busca inteligente"), nao como default.

  transformers vs LLM: a diferenca real

  "transformers" (a biblioteca) roda qualquer modelo -- desde um BERT de 110M

  modelo:

  - Encoder-only (BERT, cross-encoder) -- classifica, pontua, rankeia. Nao gera
  texto. Leve, rapido, deterministico. Roda tranquilo na VM em CPU
  - Seq2seq (T5, mT5) -- gera texto, mas precisa de fine-tuning para a tarefa.
  Um mT5-small (300M params) poderia reescrever queries se treinado pra isso.
  Mas precisaria de dataset de treinamento
  - Generativo/LLM (llama, qwen, mistral) -- gera texto de proposito geral via
  prompt. Nao precisa de treino. Mas e maior e mais lento

  O que isso significa na pratica

  A maioria das melhorias nao precisa de LLM nenhum. Regex + dicionario +
  cross-encoder cobrem muito terreno. Sao robustos, deterministicos, nao
  quebram.

  A decomposicao multi-perspectiva e o unico caso que genuinamente precisa de
  geracao de texto. E ai a escolha e:

  - Modelo pequeno fine-tuned (mT5, ~300M) -- rapido, previsivel, mas precisa de
   dados de treino que nao temos hoje
  - LLM via prompt (ollama/vLLM, ~7-8B) -- flexivel, sem treino, mas menos
  previsivel e mais pesado

  tenha que pensar em qual modo esta usando. A query em si carrega a intencao.

  Tenho contexto suficiente para propor abordagens. Vou apresentar tres:

  ---
  Abordagem 1: LLM como preprocessador (inline, transparente)

  Toda query passa pelo LLM antes da busca. O LLM detecta a intencao
  (exploratoria, com tese, adversarial), decompoe em perspectivas, e retorna
  queries + filtros. O Laravel orquestra as buscas no Rust, funde resultados.

  - Pro: transparente, o usuario so digita. Funciona para os tres modos
  - Contra: toda busca paga o custo do LLM. Queries simples ("dano moral") nao
  precisam disso

  Abordagem 2: LLM condicional (so quando vale a pena)

  Heuristica decide se a query precisa de LLM: queries curtas e genericas vao
  direto para o Rust (com o step 0 que ja temos). Queries longas, com tese
  explicita, ou com linguagem complexa passam pelo LLM primeiro.

  - Pro: economiza chamadas LLM em queries simples
  - Contra: a heuristica pode errar. Adiciona uma decisao que pode frustrar
  ("por que essa busca foi pior?")

  Abordagem 3: LLM sempre, com cross-encoder pos-busca

  LLM decompoe toda query (como abordagem 1), e apos a busca, um cross-encoder
  local reordena os resultados avaliando relevancia query-documento. Isso ataca
  o problema da direcionalidade (favoravel vs contraria a tese).

  - Pro: melhor qualidade de ranking, especialmente para busca com tese
  - Contra: mais complexo, duas camadas de modelo. Cross-encoder precisa ler
  cada resultado

  ---
  Minha recomendacao: abordagem 3. A abordagem 1 resolve decomposicao mas nao
  resolve ranking por direcionalidade. A abordagem 2 e fragil. A abordagem 3
  cobre os tres modos e ataca o problema que voce descreveu -- priorizar
  resultados que confirmam a tese.

  O cross-encoder roda local em CPU (modelo ~100MB, sem custo por query). O LLM
  (Claude CLI) roda uma vez por busca.
